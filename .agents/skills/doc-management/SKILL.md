# ドキュメント管理スキル (Documentation Management)

このプロジェクトのドキュメントが常に正確で、読みやすく、最新の状態であることを保証するためのガイドラインです。

## このスキルを使用するタイミング

以下の状況で必ず参照・適用してください：

- **新機能の追加**: 新しい機能やエンドポイントが追加された際。
- **アーキテクチャの変更**: フォルダ構造、技術スタック、認証フローなどが変更された際。
- **環境構築手順の変更**: 環境変数や依存関係、起動手順などが変わった際。
- **定期的なメンテナンス**: 定期的にドキュメントの鮮度を確認し、古くなった情報を更新する際。

## ドキュメント標準

### 1. README.md (プロジェクトの顔)
- **構成**: プロジェクトの概要、アーキテクチャ（Mermaid図含む）、環境構築手順、技術スタック、エージェントスキル。
- **更新ポリシー**: 
    - 技術スタックの変更時は即座に「技術スタック詳細」セクションを更新。
    - フォルダ構造の変更時は `tree` コマンド等で確認し、アーキテクチャ解説を修正。
    - **エージェントスキル**が追加・削除された場合は、専用セクションを更新。

### 2. walkthrough.md (開発の軌跡)
- **構成**: 実装完了した機能の一覧、実行手順、次のステップ。
- **更新ポリシー**: 
    - 各タスク（LINE認証、楽天API連携等）の完了時に、その成果と設定方法を追記。
    - 「次のステップ」は常に最新の未完了タスクを反映させる。

### 3. API 仕様 / コードコメント
- **JSDoc**: 関数やコンポーネントには JSDoc 形式で型と目的を記述する。
- **README 内の実装解説**: `README.md` に記載する実装詳細は、抽象度の高い「設計思想」に留め、具体的な仕様はコード内のコメントや別途作成する仕様書（`SPEC.md` 等）に任せる。

## トーン＆マナー (日本語)

- 言葉遣いは「です・ます」調（敬体）を基本とする。
- 技術用語は定着しているものはカタカナ表記（例: レンダリング、ハイドレーション）。
- 読みやすさを重視し、適宜箇条書きやテーブル、図（Mermaid）を活用する。

## チェックリスト

ドキュメントを更新する際は、以下の項目をセルフチェックしてください：

- [ ] パスは絶対パスではなく、プロジェクトルートからの相対パスまたはプロジェクト内のリンク形式になっているか。
- [ ] 実行コマンドは実際に動作する最小限のものが記載されているか。
- [ ] 環境変数などの秘匿情報はサンプル値に置き換えられているか。
- [ ] タイポや誤字脱字がないか。
- [ ] 日本語として自然で、開発者が理解しやすい表現になっているか。
